# Agent Infrastructure

This directory holds the terraform configuration for the agent's dedicated GCP project.

## What this manages

- All Google Cloud APIs the agent needs (Secret Manager, Vertex AI, Storage, Drive/Sheets/Docs).
- The agent's service account.
- A staging bucket for ADK deployments.
- Per-platform secret containers (Slack/Telegram/Google Chat/Discord) and the IAM bindings that grant The Forum's Cloud Run service account `secretAccessor` on each.
- (Optional) Scheduler MCP key secret container and IAM binding for the agent's Reasoning Engine SA.

## What this does NOT manage

- **The GCP project itself.** Created via `gcloud projects create` in `../get_started_linux.sh` (or manually). Terraform references it as a `data` source so you can't accidentally `terraform destroy` your entire project.
- **The terraform state bucket.** Created via `gcloud storage buckets create` in `../get_started_linux.sh`. Must exist before `terraform init`.
- **The Reasoning Engine itself.** Deployed by `../deploy_and_update.sh` via `adk deploy agent_engine`. Vertex AI Agent Engine resources aren't a clean fit for terraform — the deploy script owns their lifecycle.
- **Secret VALUES.** Terraform creates empty containers; values are populated via `gcloud secrets versions add` after apply (so they never land in terraform state). `get_started_linux.sh` handles this for you on initial setup.

## Layout

| File | Purpose |
|---|---|
| `main.tf` | Common infra (Section 1) + platform sections 2-6 (commented out by default) |
| `variables.tf` | Variable declarations with descriptions |
| `terraform.tfvars.example` | Template for your `terraform.tfvars` |
| `terraform.tfvars` | Your config (gitignored, generated by `get_started_linux.sh`) |
| `providers.tf` | Provider versions + GCS backend (backend block populated by `get_started_linux.sh`) |

## Usage

If you ran `../get_started_linux.sh`, everything below has been done for you. If you're managing infrastructure manually:

```bash
# 1. Bootstrap (one-time)
gcloud projects create YOUR_PROJECT_ID --organization=YOUR_ORG_ID
gcloud beta billing projects link YOUR_PROJECT_ID --billing-account=YOUR_BILLING
gcloud services enable serviceusage.googleapis.com cloudresourcemanager.googleapis.com \
  --project=YOUR_PROJECT_ID
gcloud storage buckets create gs://YOUR_PROJECT_ID-tfstate \
  --project=YOUR_PROJECT_ID --location=us-central1 \
  --uniform-bucket-level-access --public-access-prevention
gcloud storage buckets update gs://YOUR_PROJECT_ID-tfstate --versioning

# 2. Configure
cp terraform.tfvars.example terraform.tfvars
$EDITOR terraform.tfvars
# Edit providers.tf: uncomment the backend "gcs" block, set bucket = "YOUR_PROJECT_ID-tfstate"

# 3. Uncomment whichever platform sections (2-6) you need in main.tf

# 4. Apply
terraform init
terraform plan
terraform apply

# 5. Populate platform secret values (per platform you enabled)
# See terraform.tfvars.example for the per-platform commands.
```

## Adding a new platform later

1. Uncomment the relevant section in `main.tf`.
2. `terraform apply` (creates the secret container + IAM binding).
3. Populate the secret value via `gcloud secrets versions add` (see `terraform.tfvars.example`).
4. Re-run `../deploy_and_update.sh` — `register_agent.py` auto-detects the new secret and adds the platform to the agent's Firestore document.

## Hard rule

Never add cloud resources for this agent outside terraform. Modify `main.tf`, `terraform apply`, commit. See [`../AGENTS.md`](../AGENTS.md).
